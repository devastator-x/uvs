# 제로데이 & 신규취약점 대응 프로세스

## 1. 개요

본 문서는 제로데이 및 신규 취약점 발생 시, 정보 수집부터 전사 대응까지의 전체 프로세스를 정의한다.

| 항목 | 내용 |
|------|------|
| 문서 버전 | v1.0 |
| 최초 작성일 | 2026-03-17 |
| 작성 부서 | 정보보안팀 |
| 대상 | 정보보안팀, 인프라플랫폼팀, 전사 개발/운영 조직 |

## 2. 취약점 정보 수집

### 2.1 자동 수집 체계

취약점 자동 수집기(vuln_collector)를 통해 주요 소스로부터 신규 취약점을 실시간으로 수집한다.

| 수집 소스 | 수집 주기 | 수집 내용 |
|-----------|-----------|-----------|
| NVD (NIST) | 2시간 | CVE 등록/갱신 정보, CVSS 점수, CWE 분류, 영향 제품(CPE) |
| MITRE CVE | 2시간 | 신규 CVE 할당 정보, 변경 이력 |
| CISA KEV | 6시간 | 실제 악용이 확인된 취약점(Known Exploited Vulnerabilities) 카탈로그 |
| GitHub Security Advisories | 4시간 | 오픈소스 패키지 취약점, 패치 정보 |

### 2.2 수동 모니터링

자동 수집 외에 다음 채널을 통해 추가 정보를 모니터링한다.

- **KISA 보호나라**: 국내 보안 공지 및 업데이트 권고
- **벤더 보안 공지**: Microsoft, Apache, Oracle 등 주요 벤더 Security Advisory
- **보안 커뮤니티**: Full Disclosure, oss-security, Twitter/X 보안 연구자 채널
- **위협 인텔리전스**: Exploit-DB, Packet Storm, PoC-in-GitHub 등 PoC 공개 모니터링

### 2.3 수집 데이터 관리

- 수집된 취약점은 내부 DB에 저장되며, 한글 번역이 자동 적용된다.
- 취약점 피드 대시보드(`/vulnfeed`)를 통해 정보보안팀이 상시 모니터링한다.
- CISA KEV에 등록된 취약점은 **실제 공격에 활용 중**인 것으로 간주하여 우선 검토 대상으로 분류한다.

## 3. 취약점 심각도 분류

수집된 취약점은 CVSS 점수와 취약점 유형, 자사 영향도를 기반으로 다음과 같이 분류한다.

### 3.1 등급 기준

| 등급 | CVSS 점수 | 조건 | 대응 수준 |
|------|-----------|------|-----------|
| **긴급(Critical)** | 9.0 이상 | RCE, 인증우회, 권한상승 중 하나 이상 해당 | 즉시 대응 (4시간 이내 검토 착수) |
| **높음(High)** | 8.0 ~ 8.9 | RCE, 인증우회, 권한상승 중 하나 이상 해당 | 신속 대응 (1영업일 이내 검토 착수) |
| **높음(High)** | 7.0 ~ 8.9 | 정보유출, DoS 등 | 일반 대응 (3영업일 이내 검토) |
| **중간(Medium)** | 4.0 ~ 6.9 | - | 정기 대응 (정기 패치 사이클에 포함) |
| **낮음(Low)** | 0.1 ~ 3.9 | - | 모니터링 (리스크 수용 여부 판단) |

### 3.2 긴급도 상향 조건

다음 조건에 해당하면 등급과 무관하게 **긴급 대응**으로 상향한다.

- CISA KEV에 등록된 취약점 (실제 악용 확인)
- PoC 코드가 공개된 취약점
- 자사 인프라에서 사용 중인 제품/버전에 직접 해당하는 취약점
- 언론 보도 또는 업계 이슈로 확산 중인 취약점

## 4. 대응 프로세스

### 4.1 전체 흐름

```
취약점 수집 → 영향도 분석 → 심각도 분류 → 내부 검토 → 전사 공지 → 점검 가이드 배포 → 점검 스크립트 제작 → 인프라 점검 실행 → 조치 확인 → 완료 보고
```

### 4.2 단계별 상세

#### STEP 1. 취약점 수집 및 탐지

| 항목 | 내용 |
|------|------|
| 담당 | 정보보안팀 |
| 도구 | vuln_collector 자동 수집기, 취약점 피드 대시보드 |
| 활동 | 수집된 취약점 중 자사 영향 가능성이 있는 항목 식별 |
| 산출물 | 검토 대상 취약점 목록 |

#### STEP 2. 영향도 분석 및 내부 검토

| 항목 | 내용 |
|------|------|
| 담당 | 정보보안팀 |
| 활동 | - CVSS 점수, 취약점 유형(RCE/인증우회/권한상승 등) 확인 |
| | - 자사 인프라 및 서비스의 해당 제품/버전 사용 여부 확인 |
| | - 공격 가능성 및 비즈니스 영향도 평가 |
| | - 대응 등급 결정 (긴급/높음/중간/낮음) |
| 산출물 | 취약점 분석 보고서 (내부 검토 완료) |

#### STEP 3. 전사 공지

등급 및 조건에 따라 다음과 같이 공지한다.

| 대응 등급 | 공지 채널 | 공지 내용 |
|-----------|-----------|-----------|
| **긴급/높음** (CVSS 8.0 이상 + RCE/인증우회/권한상승) | Slack `#security-alert` 채널 전사 공지 | 취약점 개요, 영향 범위, 긴급 조치 사항 |
| **긴급/높음** (CISA KEV 등재 or PoC 공개) | Slack `#security-alert` + 관련 팀 다이렉트 | 취약점 개요, 즉시 조치 필요 사항 |
| **중간** | Slack `#security-notice` 채널 | 취약점 요약, 정기 패치 일정 안내 |
| **낮음** | 취약점 피드 대시보드 게시 | 대시보드에서 조회 가능 |

**Slack 공지 템플릿:**

```
🚨 [긴급 보안 공지] CVE-XXXX-XXXXX

▶ 취약점명: {취약점 제목}
▶ CVSS 점수: {점수} ({등급})
▶ 취약점 유형: {RCE / 인증우회 / 권한상승 등}
▶ 영향 제품: {제품명 및 영향 버전}
▶ 자사 영향 범위: {영향받는 시스템/서비스 목록}
▶ 권고 조치: {패치 적용 / 설정 변경 / 임시 완화 조치 등}
▶ 조치 기한: {YYYY-MM-DD}
▶ 점검 가이드: {Confluence 링크}
▶ 담당: 정보보안팀
```

#### STEP 4. 취약점 점검 가이드라인 배포

| 항목 | 내용 |
|------|------|
| 담당 | 정보보안팀 |
| 활동 | - 취약점 상세 분석 내용 작성 (공격 원리, 영향 범위) |
| | - 해당/비해당 판단 기준 제시 |
| | - 수동 점검 방법 안내 |
| | - 패치 적용 방법 또는 임시 완화 조치(Workaround) 안내 |
| 배포 | Confluence 보안 공간에 게시, Slack 공지에 링크 포함 |
| 산출물 | 취약점 점검 가이드라인 문서 |

#### STEP 5. 취약점 점검 스크립트 제작

| 항목 | 내용 |
|------|------|
| 담당 | 정보보안팀 |
| 활동 | - 해당 취약점의 해당/비해당 여부를 자동 판별하는 점검 스크립트 작성 |
| | - Ansible Playbook에서 호출 가능한 형태(Bash 스크립트)로 제작 |
| | - 점검 결과를 표준 출력 형식(JSON/CSV)으로 반환하도록 구현 |
| | - 내부 테스트 환경에서 검증 완료 후 전달 |
| 산출물 | 취약점 점검 스크립트 + 사용 가이드 |

**점검 스크립트 표준 출력 형식:**

```json
{
  "cve_id": "CVE-XXXX-XXXXX",
  "hostname": "server01",
  "check_date": "2026-03-17",
  "status": "vulnerable | not_vulnerable | error",
  "details": "점검 상세 결과",
  "affected_version": "설치된 버전",
  "fixed_version": "패치 버전"
}
```

#### STEP 6. 인프라 점검 실행

| 항목 | 내용 |
|------|------|
| 담당 | 인프라플랫폼팀 |
| 활동 | - 정보보안팀으로부터 전달받은 점검 스크립트를 Ansible을 통해 대상 서버에 배포 및 실행 |
| | - 점검 결과를 수집하여 정보보안팀에 전달 |
| 산출물 | 전사 서버 점검 결과 |

#### STEP 7. 조치 및 패치 적용

| 항목 | 내용 |
|------|------|
| 담당 | 인프라플랫폼팀 / 해당 서비스 담당 팀 |
| 활동 | - 취약 판정 서버에 대해 패치 적용 또는 완화 조치 수행 |
| | - 조치 완료 후 정보보안팀에 결과 보고 |
| | - 필요 시 재점검 실시 |
| 산출물 | 조치 결과 보고 |

#### STEP 8. 완료 보고 및 종결

| 항목 | 내용 |
|------|------|
| 담당 | 정보보안팀 |
| 활동 | - 전체 대응 결과 정리 |
| | - 잔여 리스크 평가 및 수용 여부 판단 |
| | - Confluence에 대응 완료 보고서 작성 |
| | - Slack `#security-alert` 채널에 조치 완료 공지 |
| 산출물 | 대응 완료 보고서 |

## 5. 역할 및 책임 (R&R)

| 단계 | 정보보안팀 | 인프라플랫폼팀 | 서비스 담당 팀 |
|------|-----------|---------------|---------------|
| 취약점 수집/탐지 | **수행** | - | - |
| 영향도 분석/내부 검토 | **수행** | 정보 제공 (자산 현황) | 정보 제공 (사용 기술 스택) |
| 전사 공지 | **수행** | - | - |
| 점검 가이드라인 작성 | **수행** | 검토 | - |
| 점검 스크립트 제작 | **수행** | 검토/피드백 | - |
| 인프라 점검 실행 | 결과 분석 | **수행** (Ansible 배포) | 애플리케이션 레벨 점검 |
| 조치/패치 적용 | 검증 | **수행** (인프라) | **수행** (애플리케이션) |
| 완료 보고 | **수행** | 조치 결과 제출 | 조치 결과 제출 |

## 7. 도구 및 시스템

| 도구 | 용도 | 비고 |
|------|------|------|
| vuln_collector | 취약점 자동 수집 (NVD, MITRE, CISA KEV, GitHub Advisories) | PM2로 운영, 2~6시간 주기 |
| 취약점 피드 대시보드 | 수집된 취약점 조회/검색/필터링 | `/vulnfeed` |
| Confluence | 점검 가이드라인, 대응 보고서 관리 | 보안팀 공간 |
| Slack | 전사 보안 공지 | `#security-alert`, `#security-notice` |
| Ansible | 점검 스크립트 대규모 배포 및 실행 | 인프라플랫폼팀 운영 |
| Git | 점검 스크립트 버전 관리 | vuln_scanner 저장소 |
