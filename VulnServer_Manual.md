# 취약점 보고서 WAS 설명서

## 1. 개요

**취약점 보고서 WAS**는 취약점 점검 Agent가 생성한 JSON 결과 파일을 업로드하여 웹 브라우저에서 시각적 보고서로 확인할 수 있는 웹 애플리케이션입니다.

| 항목 | 내용 |
|------|------|
| 경로 | `vuln-report-was/` |
| 버전 | 1.1 |
| 기술 스택 | Node.js 18 + Express + EJS + Chart.js(CDN) + Puppeteer |
| 실행 포트 | 3000 (기본값) |
| 외부 의존성 | `express`, `ejs`, `multer`, `puppeteer` |

---

## 2. 기능 요약

| 기능 | 설명 |
|------|------|
| JSON 업로드 | 드래그&드롭 또는 클릭으로 다중 파일 업로드 |
| 서버 목록 | 업로드된 보고서 목록 (호스트명, OS, 점검일, 양호/취약/N/A 건수) |
| 보고서 뷰 | 시스템 정보 + 도넛/막대 차트 + 104항목 테이블 (필터/정렬/검색) |
| PDF 다운로드 | 보고서 페이지를 그대로 PDF로 변환하여 다운로드 |
| 삭제 | 업로드된 보고서 파일 개별 삭제 |

## 3. 파일 구조

```
vuln-report-was/
├── server.js                  # Express 서버 (라우트, 헬퍼 함수)
├── package.json               # 의존성 정의
├── views/
│   ├── index.ejs              # 메인 페이지 (업로드 + 서버 목록)
│   └── report.ejs             # 보고서 페이지 (대시보드 + 차트 + 테이블)

│   └── css/
│       └── style.css          # 반응형 + 인쇄 CSS
└── uploads/                   # 업로드된 JSON 파일 저장 디렉터리 (자동 생성)
```

## 4. 설치 및 실행

### 4.1 설치

```bash
cd vuln-report-was
npm install
```

### 4.2 실행

```bash
node server.js
# 또는
npm start
```

실행 후 브라우저에서 `http://localhost:3000` 접속

### 4.3 시스템 요구사항

| 항목 | 요구사항 |
|------|----------|
| Node.js | 18.x 이상 |
| Chromium | puppeteer 설치 시 자동 포함 |
| 한글 폰트 | PDF 한글 출력 시 필요 (`fonts-noto-cjk` 권장) |
| 디스크 | puppeteer(Chromium) 설치로 약 300MB 이상 필요 |

### 4.4 한글 폰트 설치 (PDF용)

PDF에서 한글이 깨질 경우 서버에 한글 폰트를 설치합니다.

```bash
# Ubuntu/Debian
sudo apt-get install -y fonts-noto-cjk

# RHEL/CentOS/Rocky
sudo yum install -y google-noto-sans-cjk-ttc-fonts
```

## 5. 사용 방법

### 5.1 JSON 업로드

1. 메인 페이지에서 드래그&드롭 영역에 JSON 파일을 끌어놓거나 클릭하여 선택
2. 여러 파일을 동시에 선택 가능 (최대 50개)
3. "업로드" 버튼 클릭
4. 업로드 완료 후 서버 목록에 자동 추가

파일명은 `{호스트명}_{점검일}.json` 형식으로 자동 변환됩니다. 동일한 파일이 이미 존재하면 `_1`, `_2` 등의 접미사가 자동 추가됩니다.

### 5.2 보고서 열람

서버 목록에서 "보기" 버튼을 클릭하면 보고서 페이지로 이동합니다.

보고서 페이지 구성:

| 영역 | 내용 |
|------|------|
| 시스템 정보 | 호스트명, OS, 커널, IP, 점검일시 |
| 요약 카드 | 전체/취약/양호/N/A 건수 및 비율 |
| 도넛 차트 | 취약/양호/N/A 비율 시각화 |
| 막대 차트 | 카테고리별 취약/양호/N/A 분포 (가로 누적) |
| 결과 테이블 | 104개 항목 상세 (ID, 카테고리, 제목, 중요도, 결과, 상세, 현재설정) |

### 5.3 필터/정렬/검색

| 기능 | 사용법 |
|------|--------|
| 필터 | "전체", "취약", "양호", "N/A" 버튼 클릭으로 결과별 필터링 |
| 검색 | 검색 입력란에 키워드 입력 (ID, 제목, 상세 내용 대상) |
| 정렬 | 테이블 헤더(ID, 카테고리, 제목, 중요도, 결과) 클릭으로 오름차순/내림차순 전환 |

필터와 검색은 동시에 적용됩니다 (예: "취약" 필터 + "SSH" 검색).

### 5.4 PDF 다운로드

보고서 페이지 우측 상단 "PDF 다운로드" 버튼을 클릭하면, 현재 보고서 페이지가 그대로 A4 세로 PDF로 변환되어 다운로드됩니다.

- 파일명: `취약점점검보고서_{호스트명}_{점검일}.pdf`
- 용지: A4 세로, 배경색 포함
- 차트 및 테이블이 모두 PDF에 포함됩니다

### 5.5 삭제

서버 목록에서 "삭제" 버튼 클릭 → 확인 대화상자 → 삭제 완료

## 6. API 라우트

| 메서드 | 경로 | 설명 |
|--------|------|------|
| GET | `/` | 메인 페이지 (업로드 + 서버 목록) |
| POST | `/upload` | JSON 파일 업로드 (multipart/form-data, 필드명: `files`) |
| GET | `/report/:filename` | 보고서 페이지 |
| GET | `/download/:filename` | PDF 다운로드 |
| POST | `/delete/:filename` | 파일 삭제 |

### 6.1 입력 JSON 형식

업로드하는 JSON 파일은 취약점 점검 Agent(`vul_scanner.sh`)의 출력 형식을 따릅니다. 필수 필드:

```json
{
  "hostname": "server-01",
  "os": "Ubuntu",
  "os_version": "24.04",
  "kernel": "6.6.87",
  "ip_addresses": ["192.168.1.10"],
  "scan_date": "2026-03-05T16:27:40+09:00",
  "summary": { "total": 104, "pass": 49, "fail": 47, "na": 8 },
  "results": [
    {
      "id": "U-01",
      "category": "계정관리",
      "title": "root 계정 원격 접속 제한",
      "importance": "상",
      "status": "양호",
      "detail": "SSH PermitRootLogin 미설정",
      "current_settings": "PermitRootLogin=미설정"
    }
  ]
}
```

`results` 배열이 반드시 존재해야 하며, 없을 경우 업로드가 거부됩니다.

## 7. 보안

### 7.1 적용된 보안 조치

| 항목 | 조치 |
|------|------|
| Path Traversal | 모든 파일 접근 라우트에서 `isSafeFilename()` 검증 (`/`, `\`, `..` 차단) |
| XSS | EJS `<%= %>` 태그를 통한 자동 HTML 이스케이프 |
| 파일 확장자 검증 | `.json` 확장자만 업로드 허용 (multer fileFilter) |
| JSON 유효성 검증 | `results` 배열 존재 여부 확인, 파싱 실패 시 파일 삭제 |
| 파일 크기 제한 | 50MB 이하 (multer limits) |
| 에러 처리 | multer 에러 시 서버 크래시 방지, 사용자에게 에러 메시지 리다이렉트 |

### 7.2 참고사항

- 현재 인증/접근 제어 기능은 없습니다. 내부 네트워크에서만 운영하거나, 별도 리버스 프록시(Nginx 등)에서 인증을 적용하는 것을 권장합니다.
- 업로드된 JSON 파일은 `uploads/` 디렉터리에 평문으로 저장됩니다.

## 8. 아키텍처

```
┌─────────┐      JSON 업로드      ┌──────────────────────────────────┐
│ 점검     │  ─────────────────→  │        보고서 WAS (Express)       │
│ Agent   │   vul_scan_*.json    │                                  │
│ (서버)   │                      │  GET /          → index.ejs      │
└─────────┘                      │  POST /upload   → 파일 저장       │
                                 │  GET /report/:f → report.ejs     │
┌─────────┐   브라우저 접속       │  GET /download/:f → PDF 생성     │
│ 점검자   │ ←─────────────────→ │  POST /delete/:f → 파일 삭제     │
│ (사용자) │   보고서 열람/다운로드  │                                  │
└─────────┘                      ├──────────────────────────────────┤
                                 │  uploads/        JSON 파일 저장   │
                                 │  public/css/     정적 CSS         │
                                 │  views/          EJS 템플릿       │
                                 └──────────────────────────────────┘
                                          │
                                          │ PDF 생성 시
                                          ▼
                                 ┌──────────────────┐
                                 │  Puppeteer        │
                                 │  (Headless Chrome) │
                                 │  localhost:3000    │
                                 │  /report/:f 접속   │
                                 │  → PDF 변환        │
                                 └──────────────────┘
```

## 9. 운영 흐름

```
1. 대상 서버에서 점검 Agent 실행
   $ sudo bash vul_scanner.sh --output /tmp/result.json

2. 생성된 JSON 파일을 WAS에 업로드
   - 웹 UI: 드래그&드롭 또는 파일 선택
   - CLI:  curl -F "files=@result.json" http://WAS서버:3000/upload

3. 웹 브라우저에서 보고서 열람
   http://WAS서버:3000

4. 필요 시 PDF 다운로드
   - 보고서 페이지에서 "PDF 다운로드" 클릭
   - CLI:  curl -o report.pdf http://WAS서버:3000/download/파일명.json
```

## 10. FAQ

**Q: PDF에서 한글이 깨져 보입니다 (네모 박스).**
A: 서버에 한글 폰트가 설치되어 있지 않기 때문입니다. `fonts-noto-cjk` 패키지를 설치하세요 (4.4절 참조).

**Q: PDF 다운로드 시 시간이 오래 걸립니다.**
A: Puppeteer가 Chromium을 실행하여 페이지를 렌더링하므로 첫 번째 요청 시 수 초가 소요될 수 있습니다. 이는 정상 동작입니다.

**Q: 동시에 여러 사용자가 PDF를 다운로드해도 되나요?**
A: 가능합니다. 각 요청마다 별도의 Chromium 인스턴스가 생성되어 처리됩니다. 다만 서버 리소스(메모리)를 고려하여 동시 사용자가 많은 환경에서는 주의가 필요합니다.

**Q: JSON 파일이 아닌 파일을 업로드하면 어떻게 되나요?**
A: `.json` 확장자가 아닌 파일은 업로드가 거부되며, JSON 파싱이 실패하거나 `results` 배열이 없는 경우에도 에러 메시지와 함께 거부됩니다.

**Q: 업로드한 파일은 어디에 저장되나요?**
A: `vuln-report-was/uploads/` 디렉터리에 `{호스트명}_{점검일}.json` 형식으로 저장됩니다.

**Q: 인증 기능이 없는데 외부 공개해도 되나요?**
A: 권장하지 않습니다. 내부 네트워크 전용으로 사용하거나, Nginx 등의 리버스 프록시에서 Basic Auth 또는 IP 제한을 적용하세요.
